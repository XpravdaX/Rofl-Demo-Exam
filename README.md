## Приведение к 3 нормальной форме (3НФ)

### Что это вообще такое?
Это способ организации таблиц в БД, чтобы не было:
- Повторяющихся данных (трата места)
- Ошибок при обновлении (обновил в одном месте, в другом забыл)
- Странных зависимостей (когда одно неважное поле тянет за собой другое)

### Пример: "Магазин яблок"

Представим, что у нас есть таблица с продажами яблок:

**Исходная таблица (сырые данные):**

| Заказ | Покупатель | Город | Товары | Цена за кг | Кг | Сумма |
|-------|------------|-------|--------|------------|-----|-------|
| 101 | Иван | Москва | Яблоки Антоновка, Яблоки Гренни Смит | 150, 200 | 2, 1 | 500 |
| 102 | Петр | Питер | Яблоки Антоновка | 150 | 3 | 450 |
| 103 | Иван | Москва | Яблоки Гренни Смит | 200 | 2 | 400 |

**Проблемы:**
- В одной ячейке несколько значений через запятую
- Названия сортов повторяются
- Цена яблок хранится в заказе (а если цена изменится?)

---

### По шагам:
#### 1НФ (Первая нормальная форма) "В каждой ячейке одно значение"

**Проблема:** В ячейках "Товары", "Цена за кг", "Кг" списки через запятую.

**Как исправить:** Разбиваем на отдельные строки (каждый товар в отдельной строке)

| Заказ | Покупатель | Город | Товар | Цена за кг | Кг | Сумма |
|-------|------------|-------|-------|------------|-----|-------|
| 101 | Иван | Москва | Яблоки Антоновка | 150 | 2 | 300 |
| 101 | Иван | Москва | Яблоки Гренни Смит | 200 | 1 | 200 |
| 102 | Петр | Питер | Яблоки Антоновка | 150 | 3 | 450 |
| 103 | Иван | Москва | Яблоки Гренни Смит | 200 | 2 | 400 |

**Сделали:** В каждой клетке теперь одно значение

---

#### 2НФ (Вторая нормальная форма) "Всё зависит от ключа полностью"

**Проблема:** Ключ тут составной (Заказ + Товар). Но "Покупатель" и "Город" зависят только от Заказа (от части ключа), а не от Товара. Из-за этого они повторяются.

**Как исправить:** Выносим информацию о заказе в отдельную таблицу

**Таблица "Заказы" (головная):**

| Заказ ID | Покупатель | Город |
|----------|------------|-------|
| 101 | Иван | Москва |
| 102 | Петр | Питер |
| 103 | Иван | Москва |

**Таблица "Состав заказа" (детальная):**

| Заказ ID | Товар | Цена за кг | Кг | Сумма |
|----------|-------|------------|-----|-------|
| 101 | Яблоки Антоновка | 150 | 2 | 300 |
| 101 | Яблоки Гренни Смит | 200 | 1 | 200 |
| 102 | Яблоки Антоновка | 150 | 3 | 450 |
| 103 | Яблоки Гренни Смит | 200 | 2 | 400 |

**Сделали:** Данные о покупателе хранятся один раз, не дублируются к каждому товару

---

#### 3НФ (Третья нормальная форма) "Нет зависимостей между неключевыми полями"

**Проблема:** В таблице "Заказы" поле "Город" зависит от "Покупателя". Если Иван переедет в Питер, придется менять город во всех его заказах.

**Как исправить:** Создаем таблицу покупателей, а в заказах оставляем только ссылку (ID)

**Таблица "Покупатели":**

| Покупатель ID | Имя | Город |
|---------------|-----|-------|
| 1 | Иван | Москва |
| 2 | Петр | Питер |

**Таблица "Заказы" (обновленная):**

| Заказ ID | Покупатель ID |
|----------|---------------|
| 101 | 1 |
| 102 | 2 |
| 103 | 1 |

**Таблица "Состав заказа" (без изменений):**

| Заказ ID | Товар | Цена за кг | Кг | Сумма |
|----------|-------|------------|-----|-------|
| 101 | Яблоки Антоновка | 150 | 2 | 300 |
| 101 | Яблоки Гренни Смит | 200 | 1 | 200 |
| 102 | Яблоки Антоновка | 150 | 3 | 450 |
| 103 | Яблоки Гренни Смит | 200 | 2 | 400 |

---

### Но это еще не всё! Идем дальше...

**Проблема:** В таблице "Состав заказа" название товара и цена повторяются. Если цена на Антоновку вырастет до 160, придется обновлять во всех строках.

#### Продолжаем нормализацию (выходим на идеальную структуру):

**Таблица "Товары":**

| Товар ID | Название | Цена за кг |
|----------|----------|------------|
| 1 | Яблоки Антоновка | 150 |
| 2 | Яблоки Гренни Смит | 200 |

**Таблица "Состав заказа" (финальная):**

| Заказ ID | Товар ID | Кг | Сумма |
|----------|----------|-----|-------|
| 101 | 1 | 2 | 300 |
| 101 | 2 | 1 | 200 |
| 102 | 1 | 3 | 450 |
| 103 | 2 | 2 | 400 |

**Таблица "Заказы" (уже готова):**

| Заказ ID | Покупатель ID |
|----------|---------------|
| 101 | 1 |
| 102 | 2 |
| 103 | 1 |

**Таблица "Покупатели":**

| Покупатель ID | Имя | Город |
|---------------|-----|-------|
| 1 | Иван | Москва |
| 2 | Петр | Питер |

**Важно:** После нормализации рисуем ER-диаграмму
>
> **Краткая памятка:**
> 1. **1НФ** все атрибуты атомарны (нет повторяющихся групп, каждая ячейка содержит одно значение)
> 2. **2НФ** 1НФ + нет частичной зависимости от составного ключа (все неключевые поля зависят от полного первичного ключа)
> 3. **3НФ** 2НФ + нет транзитивных зависимостей (неключевые поля не зависят от других неключевых полей)

## SQL запросы на создание БД

### Создание базы данных
```sql
-- Создать новую базу данных
CREATE DATABASE shop_db;

-- Выбрать базу для работы
USE shop_db;
```
### Создание таблиц (по примеру с яблоками)
```sql
-- Таблица "Покупатели"
CREATE TABLE buyers (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL,
    city VARCHAR(100) NOT NULL
);

-- Таблица "Товары"
CREATE TABLE products (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(200) NOT NULL,
    price_per_kg DECIMAL(10, 2) NOT NULL
);

-- Таблица "Заказы"
CREATE TABLE orders (
    id INT PRIMARY KEY AUTO_INCREMENT,
    buyer_id INT NOT NULL,
    order_date DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (buyer_id) REFERENCES buyers(id)
);

-- Таблица "Состав заказа"
CREATE TABLE order_items (
    order_id INT NOT NULL,
    product_id INT NOT NULL,
    kg DECIMAL(10, 2) NOT NULL,
    amount DECIMAL(10, 2) NOT NULL,
    PRIMARY KEY (order_id, product_id),
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (product_id) REFERENCES products(id)
);
```

### Полезные команды при создании
```sql
-- Удалить таблицу (если нужно пересоздать)
DROP TABLE IF EXISTS order_items;
DROP TABLE IF EXISTS orders;
DROP TABLE IF EXISTS products;
DROP TABLE IF EXISTS buyers;

-- Показать структуру таблицы
DESCRIBE buyers;

-- Добавить столбец в существующую таблицу
ALTER TABLE buyers ADD phone VARCHAR(20);

-- Изменить тип данных столбца
ALTER TABLE products MODIFY price_per_kg DECIMAL(12, 2);
```

## SQL запросы на выборку (BETWEEN, JOIN)
### Заполнение тестовыми данными
``` sql
-- Добавляем покупателей
INSERT INTO buyers (name, city) VALUES
('Иван', 'Москва'),
('Петр', 'Питер');

-- Добавляем товары
INSERT INTO products (name, price_per_kg) VALUES
('Яблоки Антоновка', 150),
('Яблоки Гренни Смит', 200);

-- Добавляем заказы
INSERT INTO orders (buyer_id) VALUES
(1), (2), (1);

-- Добавляем состав заказов
INSERT INTO order_items (order_id, product_id, kg, amount) VALUES
(1, 1, 2, 300),
(1, 2, 1, 200),
(2, 1, 3, 450),
(3, 2, 2, 400);
```
### BETWEEN (диапазон значений)
``` sql
-- Выбрать товары с ценой от 100 до 180
SELECT * FROM products 
WHERE price_per_kg BETWEEN 100 AND 180;

-- Выбрать заказы с суммой от 250 до 500
SELECT * FROM order_items 
WHERE amount BETWEEN 250 AND 500;

-- Выбрать заказы за определенный период
SELECT * FROM orders 
WHERE order_date BETWEEN '2026-03-01' AND '2026-03-31';

-- BETWEEN с текстом (по алфавиту)
SELECT * FROM products 
WHERE name BETWEEN 'А' AND 'Я';
```

### INNER JOIN (только совпадающие записи)
``` sql
-- Получить все заказы с информацией о покупателях
SELECT orders.id, buyers.name, buyers.city, orders.order_date
FROM orders
INNER JOIN buyers ON orders.buyer_id = buyers.id;

-- Получить состав заказов с названиями товаров
SELECT order_items.order_id, 
       products.name AS product_name,
       order_items.kg, 
       order_items.amount
FROM order_items
INNER JOIN products ON order_items.product_id = products.id;
```
### LEFT JOIN (все записи из левой таблицы)
```sql
-- Все покупатели и их заказы (даже если заказов нет)
SELECT buyers.name, orders.id AS order_id, orders.order_date
FROM buyers
LEFT JOIN orders ON buyers.id = orders.buyer_id;

-- Все товары и их продажи (даже если товар не продавался)
SELECT products.name, order_items.kg, order_items.amount
FROM products
LEFT JOIN order_items ON products.id = order_items.product_id;
```
### Сложные JOIN (несколько таблиц)
```sql
-- Полная информация о заказах
SELECT 
    orders.id AS order_number,
    buyers.name AS buyer_name,
    buyers.city,
    products.name AS product_name,
    order_items.kg,
    order_items.amount,
    orders.order_date
FROM orders
INNER JOIN buyers ON orders.buyer_id = buyers.id
INNER JOIN order_items ON orders.id = order_items.order_id
INNER JOIN products ON order_items.product_id = products.id;
```
### JOIN с условиями
``` sql
-- Заказы москвичей с суммой больше 250
SELECT 
    buyers.name,
    buyers.city,
    orders.id AS order_id,
    order_items.amount
FROM buyers
INNER JOIN orders ON buyers.id = orders.buyer_id
INNER JOIN order_items ON orders.id = order_items.order_id
WHERE buyers.city = 'Москва' 
  AND order_items.amount > 250;
```
### JOIN + BETWEEN вместе
```sql
-- Товары, которые продавались в определенном диапазоне цен
SELECT DISTINCT products.name, products.price_per_kg
FROM products
INNER JOIN order_items ON products.id = order_items.product_id
WHERE products.price_per_kg BETWEEN 140 AND 210;

-- Заказы за март с информацией о покупателях
SELECT 
    buyers.name,
    buyers.city,
    orders.id AS order_id,
    orders.order_date
FROM buyers
INNER JOIN orders ON buyers.id = orders.buyer_id
WHERE orders.order_date BETWEEN '2026-03-01' AND '2026-03-31';
```

## Git Bash шпаргалка

> Команды, которые точно пригодятся на экзамене

### Настройка и старт
```bash
# Настройка имени и почты (если не делал)
git config --global user.name "Твоё Имя"
git config --global user.email "твой@email.ru"
```
### Инициализация и первый коммит
```bash
# 1. Инициализация нового репозитория

git init

# 2. Проверка статуса (какие файлы изменены/не отслеживаются)

git status

# 3. Добавление файлов в отслеживание

git add .                          # все файлы
# или точечно(Пример)
git add README.md script.sql Program.cs

# 4. Создание коммита (фиксация изменений)

git commit -m "Первый коммит: структура БД и базовый код"

# 5. Просмотр истории коммитов

git log --oneline                   # кратко
git log                             # подробно
```

### Работа с удаленным репозиторием
```bash
# 6. Привязка к удаленному репозиторию (GitHub / GitLab)

git remote add origin https://github.com/username/project.git

# 7. Проверка удаленных репозиториев

git remote -v

# 8. Отправка изменений на сервер (в ветку main/master)

git push -u origin main              # первый раз с флагом -u
# или если ветка называется master:
git push -u origin master

# 9. Для последующих коммитов (после изменений)

git add .
git commit -m "Описание изменений"
git push                             # без флагов
```

## Мои заметки

> *Сюда буду записывать важные моменты во время подготовки и на экзамене*
### Важные моменты, чтобы не слить баллы:
#### 1. Результат ER-диаграммы в PDF формате
- **Обязательно** сохраняю финальную ER-диаграмму в **PDF**
- Проверяю, чтобы были видны:
  - Все таблицы
  - Связи (один-ко-многим, многие-ко-многим)
  - Первичные и внешние ключи
- **На ER-диаграмму выделяю не больше 30 минут**

#### 2. Минимальная работоспособная структура
- Сначала делаю так, чтобы **все работало и соединялось**
- Потом, если остается время, можно добавлять "плюшки" (индексы, триггеры, ограничения)
- Главное на экзамене, чтобы запросы выполнялись и структура была логичной

#### 3. В отдельные справочники ничего не добавлять, если мало времени
- Если время поджимает, **не лезу в мелочи**
- Отдельные справочники (типа списка городов, статусов) это круто, но не критично
- Лучше иметь 5–6 основных таблиц с чистыми связями, чем 10 таблиц, где половина не используется
- **Принцип:** сначала каркас, потом детали

#### 4. Вместо `name` лучше использовать `title`
- В таблицах, где хранятся названия (товаров, категорий, заказов и т.д.), называю поле **`title`**, а не `name`
- Почему? Потому что так принято во многих стандартах, и эксперт это оценит
